# 6A Workflow

## When to use

Use when the task is clearly about building a new page, module, feature, or system from scratch.

## Activation

When the user input starts with `@6A` or `6A`, output exactly:

> 6A工作流已激活
> 当前阶段：Align（需求对齐）

## Required documents

Create or update these files under `docs/YYYY_MM_DD_中文任务名_vN/` in the active project root. The parent directory carries the task name; file names use the workflow stage meaning:

- `01_ALIGNMENT_需求对齐.md`
- `02_CONSENSUS_共识确认.md`
- `03_DESIGN_架构设计.md`
- `04_TASK_任务拆分.md`
- `05_APPROVE_审批确认.md`
- `06_ACCEPTANCE_验收记录.md`
- `07_FINAL_最终报告.md`
- `08_TODO_待办清单.md`

## Stage gates

- Align: analyze project context, clarify scope, record assumptions, create alignment and consensus docs.
- Architect: design architecture, module boundaries, interfaces, data flow, and error handling.
- Atomize: split into atomic tasks with input/output contracts, dependencies, and independent validation.
- Approve: review completeness, consistency, feasibility, risk, complexity, and testability.
- Automate: implement tasks in dependency order, write or update tests, run validation, and keep docs synchronized.
- Assess: verify acceptance criteria, update acceptance docs, write final report, and create a precise TODO file.

## Rules

- use the current date and append a bundle version, for example `docs/2026_04_16_首页优化_v1/`
- if the same feature is rerun, increment the bundle version to `v2`, `v3`, and so on
- initialize the doc skeleton before implementation
- output the document bundle and wait for explicit user confirmation before generating or modifying code
- prefer decisions based on existing project patterns and local similar code
- ask the user when a decision is risky, preference-heavy, externally dependent, or unsafe to infer
- follow existing project style, tools, libraries, components, and error handling
- put secrets in `.env` or the project secret mechanism and do not commit them
- prefer test-first implementation when the project has a usable test setup
- cover normal flow, boundary conditions, and error cases
- any code-changing stage must document target file paths, line ranges, and before/after code context
- if approval fails, return to design or task breakdown before executing
