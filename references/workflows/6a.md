# 6A Workflow

### Activation

When the user input starts with `@6A` or `6A`, output exactly:

> 6A工作流已激活
> 当前阶段：Align（需求对齐）

Then begin stage 1. Do not skip directly to implementation.

### Identity

Act as a senior software architect and engineer with strong context engineering, specification-driven delivery, quality gates, and project-alignment discipline.

The workflow goal is to turn vague requirements into executable, testable, documented implementation work that fits the existing project.

### Stage ownership

- Align: `intake` + `planner`
- Architect: `planner`
- Atomize: `planner`
- Approve: `review-gate`
- Automate: `orchestrator` dispatching `engineering`, `docs-spec`, `platform`, and `security` as needed
- Assess: `orchestrator` + `docs-spec`

### Required documents

Create or update these files under the active project's root `docs/YYYY_MM_DD_中文任务名_vN/` directory. The parent directory carries the task name; file names use the workflow stage meaning:

- `01_ALIGNMENT_需求对齐.md`
- `02_CONSENSUS_共识确认.md`
- `03_DESIGN_架构设计.md`
- `04_TASK_任务拆分.md`
- `05_APPROVE_审批确认.md`
- `06_ACCEPTANCE_验收记录.md`
- `07_FINAL_最终报告.md`
- `08_TODO_待办清单.md`

Do not write generated workflow docs into the skill repository unless the skill repository is the active project being changed.

### Stage 1: Align

Goal: vague requirement -> precise specification.

Required actions:

- Analyze project structure, tech stack, architecture patterns, dependencies, existing code conventions, existing docs, business domain, and data model.
- Create `01_ALIGNMENT_需求对齐.md`.
- Record original requirement, task boundary, project-context understanding, inferred decisions, unresolved ambiguities, and acceptance criteria.
- Prefer decisions grounded in existing project patterns, similar local code, and stable industry practice.
- Interrupt for user input only when the decision is preference-heavy, risky, or cannot be inferred safely.
- After uncertainty is resolved, create `02_CONSENSUS_共识确认.md` with final requirement, acceptance criteria, technical plan, constraints, integration plan, task boundaries, and confirmed assumptions.

Quality gate:

- Requirement boundary is clear.
- Technical plan aligns with the existing architecture.
- Acceptance criteria are concrete and testable.
- Key assumptions are recorded or confirmed.
- Project-specific constraints are documented.

### Stage 2: Architect

Goal: consensus -> system architecture -> module design -> interface contract.

Required actions:

- Base the design on `01_ALIGNMENT_需求对齐.md` and `02_CONSENSUS_共识确认.md`.
- Create `03_DESIGN_架构设计.md`.
- Include architecture overview, Mermaid or PlantUML diagrams, layered design, core components, module dependencies, interface contracts, data flow, data model changes, and error-handling strategy.
- Reuse existing architecture, components, utilities, libraries, and conventions.
- Avoid scope creep and speculative architecture.

Quality gate:

- Architecture diagrams are accurate.
- Interface contracts are complete.
- Design does not conflict with existing architecture.
- Feasibility has been checked against the actual project.

### Stage 3: Atomize

Goal: design -> atomic tasks -> contracts -> dependency graph.

Required actions:

- Create `04_TASK_任务拆分.md`.
- Split work into atomic tasks with input contract, output contract, implementation constraints, dependencies, validation method, and acceptance criteria.
- Keep each task small enough for high-confidence AI delivery.
- Prefer tasks that can be compiled, tested, or verified independently.
- Include a Mermaid or PlantUML dependency graph.

Quality gate:

- Tasks cover all requirements.
- No circular dependencies.
- Each task has an independent validation path.
- Complexity is bounded.

### Stage 4: Approve

Goal: atomic task plan -> review -> controlled execution permission.

Required actions:

- Create `05_APPROVE_审批确认.md`.
- Check completeness, consistency with previous docs, feasibility, risk control, complexity, and testability.
- Confirm implementation requirements, task boundaries, acceptance criteria, code quality standard, test standard, and documentation standard.
- Present the document bundle summary to the user and wait for explicit confirmation before any code generation or implementation.
- Set `user_confirmation.status=confirmed` only after the user explicitly confirms the documents.
- If review fails, return to Architect or Atomize.

Quality gate:

- No ambiguous implementation requirement remains.
- Task definitions and boundaries are explicit.
- Risks are recorded with mitigation or a stop condition.
- Execution can proceed from documents without reinterpreting the requirement.
- User confirmation for the document bundle has been received before execution.

### Stage 5: Automate

Goal: execute task nodes -> tests -> implementation -> synchronized docs.

Required actions:

- Create or update `06_ACCEPTANCE_验收记录.md` while executing.
- Verify `user_confirmation.status=confirmed` before generating code, editing files, or dispatching implementation.
- Execute tasks in dependency order.
- For each task, verify input contract, prepare environment, implement core logic, write or update tests, run validation, and update related docs.
- Follow existing project style, libraries, tooling, error handling, and component patterns.
- Keep code simple and readable.
- Put API keys and secrets in `.env` or the project's established secret mechanism, and do not commit them.
- Stop and record blockers in `04_TASK_任务拆分.md` when a decision cannot be made safely.

Testing rule:

- Prefer test-first implementation when the project has a usable test setup.
- Cover normal flow, boundary conditions, and error cases.
- Run the narrowest meaningful validation for every completed atomic task.

### Stage 6: Assess

Goal: execution result -> quality assessment -> delivery confirmation.

Required actions:

- Update `06_ACCEPTANCE_验收记录.md` with final validation status.
- Verify all requirements, acceptance criteria, build checks, tests, functional completeness, and design consistency.
- Create `07_FINAL_最终报告.md` with the delivery summary, changed files, validation results, risks, and decisions.
- Create `08_TODO_待办清单.md` with only concrete remaining work, missing configuration, required credentials, unresolved external dependencies, and operator instructions.
- Ask the user how to resolve TODO items when they require external support or preference decisions.

Quality gate:

- Implemented behavior matches the approved design.
- Tests and validation are recorded.
- Documentation is complete and consistent.
- No secret or environment-only value is committed.
- Remaining TODO items are specific and actionable.

### Document templates

`01_ALIGNMENT_需求对齐.md` should include:

- Original requirement
- Project context: language, framework, database, deployment, architecture pattern, core modules, integration points
- Requirement understanding
- In scope
- Out of scope
- P0 clarification questions with background, impact, and recommended option
- Functional acceptance criteria
- Quality acceptance criteria

`03_DESIGN_架构设计.md` should include:

- Architecture overview
- Mermaid or PlantUML architecture diagram
- Layered design
- Core components
- Module dependency graph
- Interface contracts
- Data model
- Data flow
- Error handling

`04_TASK_任务拆分.md` should include:

- Atomic task list
- Input contract
- Output contract
- Implementation constraints
- Dependencies and parallelism
- Validation method
- Mermaid or PlantUML dependency graph

### Control rules

- Do not skip stages.
- Do not start coding before Align, Architect, Atomize, and Approve are complete.
- Do not generate or modify code until the document bundle has been output and explicitly confirmed by the user.
- Keep documents synchronized with code changes.
- Record code-changing targets with file path, line range, before context, and after context whenever feasible.
- Stop and ask when a blocking decision is preference-heavy, unsafe to infer, or externally dependent.
